Checkpoint Skills

TL;DR — 긴 세션의 context rot와 비대해지는 handoff 문서 문제를 해결합니다. 작업을 DAG 그래프로 저장해 새 세션에서 필요한 노드 하나만 로드합니다. PostCompact 후 자동으로 review를 주입하는 Hook 4개가 함께 동작하고, /checkpoint-viz로 그래프 전체를 인터랙티브 HTML 대시보드로 시각화할 수 있습니다.

Context Rot이란

LLM은 입력이 길어질수록 성능이 떨어집니다. 단순히 "조금 느려진다"가 아니라, 초반 토큰과 후반 토큰을 동등하게 다루지 못하고 비선형적으로 정확도가 하락합니다.

🔬
Chroma의 실험 연구 Context Rot에 따르면, "10,000번째 토큰도 100번째 토큰만큼 안정적으로 처리된다"는 가정은 실험적으로 성립하지 않습니다. NIAH(needle-in-a-haystack) 변형, LongMemEval, 반복 단어 재현 등 세 가지 실험 모두에서 입력 길이가 늘어날수록 모델 정확도가 떨어졌습니다.

코딩 에이전트와 협업할 때 context rot은 다음과 같은 형태로 나타납니다:

증상설명
초반 정보 망각세션 초반의 설계·제약사항이 토큰 뒷부분으로 갈수록 옅어져, 이미 거부했던 방향을 다시 제안하거나 합의된 규칙을 어김
검색 정확도 하락긴 입력 속 특정 정보를 찾아내는 능력이 떨어짐 — NIAH 실험에서 입력이 길어질수록 정답 검색률이 비선형으로 감소
맥락 간섭여러 주제·파일·시도가 한 context에 누적되면 서로 간섭해, 엉뚱한 파일을 수정하거나 다른 주제의 가정을 잘못 적용
일관성 저하같은 질문에 다른 답을 내거나, 앞서 작성한 코드 패턴을 다음 응답에서 깨뜨림

기존 해결 방법과 한계

Context rot을 완화하려는 시도는 이미 여럿 있습니다. 각 접근법이 무엇을 해결하고 어디서 막히는지 정리했습니다.

세 방법 모두 일부 측면은 해결하지만, 세션을 여러 번 넘기면서 방향이 바뀌고 compact가 반복되는 장기 작업에서는 빈틈이 남습니다.

🔗
보완적 도구 — Superpowers처럼 subagent로 작업을 격리해 메인 세션의 context를 보호하는 접근은 한 세션 안에서 효과적이고, Checkpoint와는 풀려는 문제가 달라 함께 써도 충돌하지 않습니다. Checkpoint는 세션 경계를 넘어가는 handoff·재개 비용 자체를 줄이는 데 집중합니다.

Checkpoint의 접근

작업을 DAG(방향 비순환 그래프) 노드로 쪼개 파일 시스템에 저장합니다. 새 세션에서는 index.md만 읽고 실행 가능한 노드 1개와 그 의존성의 Output Contract만 로드합니다.

세 가지 핵심 메커니즘으로 구성됩니다:

메커니즘 1
노드 단위 분리 저장
완료된 작업은 Output Contract(결과 요약)로 축소돼 노드 파일에 영구 보존됩니다. 거부된 옵션·실패한 시도·중간 토론은 세션과 함께 사라집니다. 재개 시에는 작업할 노드와 의존성의 Contract만 로드합니다.
메커니즘 2
후속 작업의 갈래 분리
/checkpoint-plan이 남은 작업을 종류별로 독립 노드로 만들고 의존성 그래프로 연결합니다. 어느 갈래를 작업하든 그 노드에 필요한 맥락만 정확히 주입돼, 무관한 후속 작업의 토론이 간섭하지 않습니다.
메커니즘 3
Hook 기반 자동 기록 + 점진적 재계획
PreCompact/PostCompact hook이 압축 직전 작업 흐름을 rollup으로 구조화 보존하고, 압축 후 review를 자동 주입합니다. plan은 이 기록을 읽고 지금 시점의 현실을 반영한 노드만 새로 생성합니다.

6개 스킬

위 메커니즘은 6개 슬래시 커맨드로 사용합니다. /checkpoint-review는 Compact 후 hook이 자동으로 안내하므로 직접 호출하지 않아도 됩니다.

기존 방법과의 차이

위 세 메커니즘이 앞서 본 Compact / HANDOFF.md / GSD Milestone의 한계와 각각 어떻게 대응되는지 정리합니다.

기존 방법남는 빈틈대응되는 Checkpoint 메커니즘
Compact 손실 압축이라 무엇이 사라졌는지 모름. 반복될수록 초반 결정이 옅어짐 메커니즘 3 — hook이 압축 전 흐름을 구조화 보존, 압축 후 review 자동 주입
HANDOFF.md 대화 전체를 한 문서에 통째로 옮김. 재개 시 파일 전체 로드 + 누적 메커니즘 1 — 결과만 Output Contract로 축소, 필요한 노드만 선택 로드
HANDOFF.md 후속 작업 갈래를 한 문서에 모아둬, 어느 갈래를 작업해도 다른 갈래 맥락이 따라옴 메커니즘 2 — /checkpoint-plan이 갈래를 독립 노드로 분리
GSD Milestone 사전 전체 설계가 실제와 괴리. 방향 전환 시 milestone 단위 재설계 메커니즘 3 — 실행 결과를 반영한 점진적 재계획, 현재 노드 이후만 재설계

실제 시나리오: 인증 시스템을 여러 서비스에 추가

auth-service 작업을 마치고 나면 후속 작업이 한 갈래가 아니라 여러 갈래로 나뉩니다 — user-service, api-gateway, admin-panel. 각 갈래는 서로 다른 파일·관심사를 다루고, 필요한 맥락도 다릅니다. 이 지점이 두 방식의 차이가 가장 크게 벌어지는 곳입니다.

Hook 자동화

스킬과 Hook을 함께 설치하면 사용자가 신경 쓰지 않아도 됩니다. Context rot이 위험 수준에 도달했을 때 Claude가 먼저 알려주고 /checkpoint-save를 제안합니다. 세션을 닫을 타이밍을 사용자가 매번 판단할 필요가 없습니다.

위험 수준은 어떻게 판단하나

/checkpoint-review 스킬이 다음 세 가지 신호를 종합해 위험 수준을 판단합니다 — 세션 길이, compact 누적 횟수, 그리고 프롬프트 흐름. 앞의 두 신호는 단순 카운트이지만, 핵심은 세 번째인 프롬프트 흐름입니다.

📜
프롬프트 흐름
Hook이 UserPromptSubmit 시점마다 사용자 프롬프트의 해시·길이·앞뒤 일부 텍스트를 events.jsonl에 누적해둡니다. Review는 이 기록을 통째로 읽어 지금까지의 대화가 어떤 흐름이었는지 재구성합니다.

이 흐름에서 다음과 같은 패턴이 감지되면 위험 신호로 간주합니다:
  • 주제 전환 — 코드 구현 → 디버깅 → 다른 기능 등 관심사가 바뀌는 지점
  • 작업 분기 — 한 작업 도중 발견된 다른 갈래(예: "auth 작업 중 user-service exp 조정 필요" 발견)
  • 반복·되돌이 — 같은 영역을 여러 번 오가는 흐름 (context rot 가능성)
  • 완료 지점 — 한 작업의 자연스러운 마무리 (노드를 끊기 좋은 경계)
단순 토큰 카운트로는 잡히지 않는 "지금이 세션을 끊고 노드를 정리하기 좋은 시점인가"를 의미 기반으로 판단할 수 있게 해주는 핵심 신호입니다.
🤖
Compact가 발생하면 Hook이 자동으로 점검 필요 상태를 표시하고, 다음 사용자 메시지 차례에 Claude가 /checkpoint-review를 자체 실행해 위험하면 /checkpoint-save를 제안합니다. 사용자는 평소처럼 작업하기만 하면 됩니다.

Day 1 · 마감 완료된 작업을 노드로 저장
/checkpoint-save
현재 세션에서 실제로 완료된 작업만 Output Contract로 압축해 그래프에 기록합니다. 거부된 OAuth 토론·실패 시도는 세션과 함께 사라집니다.
node-a · auth-service · Done ├ Output Contract: │ "JWT, exp 1h, refresh token 사용. │ /v1/auth/* 라우트 추가." └ 잔여: 테스트 3개 미작성
3
Day 1 · 마감 남은 작업을 갈래별 노드로 분리 핵심 단계
/checkpoint-plan
events.jsonl의 프롬프트 흐름과 save 메모를 읽어 남은 작업을 종류별로 독립 노드로 만듭니다. 다음 세션에서 어느 갈래를 작업하든 그 노드에 필요한 맥락만 정확히 주입됩니다.
node-b · user-service · Ready (deps: node-a) node-c · admin-panel · Ready (deps: node-a) node-d · api-gateway · Ready (deps: node-a) node-e · auth 테스트 · Ready (deps: node-a)

이후 /clear로 세션 닫음. 저장된 그래프는 다음 세션이 언제든 이어받을 수 있는 상태가 됩니다.

4
Day 2 · 새 세션 user-service 노드 선택 + 최소 context 로드
/checkpoint-next
빈 세션에서 시작. index.md + node-b.md + node-a의 Output Contract만 로드됩니다. admin-panel·api-gateway·테스트 노드 내용은 들어오지 않습니다.
로드된 context: ├ index.md ├ nodes/node-b.md (user-service 작업 정의) └ nodes/node-a.md → Output Contract만 (admin-panel·api-gateway 노드는 로드 안 됨) → user-service 작업 시작

작업이 끝나면 다시 /checkpoint-save → /checkpoint-plan(필요 시) → /clear 사이클 반복.

5
Day 3 · 다른 갈래 admin-panel 노드 작업 — 다른 갈래의 토론은 없음
/checkpoint-next node-c
또 빈 세션. node-c.md + node-a의 Output Contract만 로드. Day 2의 user-service 구현 토론은 다른 노드의 일이라 들어오지 않습니다.
로드된 context: ├ nodes/node-c.md (admin-panel 작업 정의) └ nodes/node-a.md → Output Contract만 (node-b user-service 토론은 로드 안 됨) → admin-panel 작업 시작 · context 부담 = Day 1과 동일
⟳
실제로 매 세션 반복되는 사이클은 Hook 알림 → review → save → plan → clear → next 다섯 단계입니다. 사용자가 직접 신경 쓰는 명령어는 /checkpoint-save, /checkpoint-plan, /checkpoint-next 세 개뿐이고, 나머지는 Hook이 자동으로 처리합니다.
